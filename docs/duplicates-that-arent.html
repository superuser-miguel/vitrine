The duplicate that was one file

Vitrine's Find Duplicates feature reported 41,727 duplicated images on a library that did not have anywhere near that many. Working out why took an argument about what a duplicate even is, two five-minute experiments, one screenshot of the app lying, and a fix that was ultimately a single field on a query.

July 2026 · SQLite · GIO · Flatpak portals · product design

This is a follow-up to What the document portal actually does. You don't need to have read it, but the short version matters here: a sandboxed app that opens a folder through the file chooser sees it under an opaque handle like /run/user/1000/doc/9bf3a2/trip/, not under its real path — and if the app indexes what it sees, the same file can end up in the database twice under two unrelated names.

1What Find Duplicates is

Two definitions of "same", both cheap by construction.

Vitrine's background index stores a BLAKE3 content hash for every file, and a perceptual hash (pHash) once enrichment has visited it. Duplicates fall out of the index without touching the image bytes again:

  • Exact — group rows by content hash. Byte-identical files, full stop.
  • Similar — union-find over pHash Hamming distance, with a BK-tree so a big library doesn't cost a billion pairwise comparisons. Catches re-encodes, resizes, minor edits.

Each cluster renders as a card: the suggested keeper, a thumbnail strip, the reclaimable bytes, and a button that trashes every copy but the keeper. That button is the loaded gun in this story — remember it.

241,727 duplicates that weren't

The hash said "same content". Nobody asked "same file?"

On the real library the exact scan lit up 41,727 content hashes with more than one present row. The library has genuine duplicates — download folders always do — but not that many. The bulk looked like this:

~/Pictures/trip/IMG_1007.jpg
/run/user/1000/doc/9bf3a2/trip/IMG_1007.jpg

One file. The first row came from browsing the folder under the app's one static grant (--filesystem=xdg-pictures); the second from the same folder being opened through the file-chooser portal at some point. Same bytes, two paths, two index rows — and a "duplicate pair" whose second copy is an illusion.

The tempting fix is to collapse the files table on content hash. It's wrong, and the reason is worth stating precisely, because it's the sentence the whole feature hangs on:

The content hash answers "are these the same bytes?". Only a resolved real path can answer "are these the same file?". Find Duplicates exists precisely to tell those two questions apart — so it can't be built on a table that has already confused them.

Two questions, two keys. Annotations (tags, ratings, comments) key on the content hash and are correct — the same image reached two ways should share its stars. The files table keys on path and is also correct — it records where things are. The bug was only ever in the seam: a feature that read "two rows, same hash" as "two copies on disk".

3The guard: prefer the durable path

Thirty minutes, no schema change, no D-Bus.

The structural fix — resolving every portal handle to its real path via the portal's Info() call and storing it as a second key — is a day of work that was deliberately deferred, because a much smaller observation covers the dangerous case: a pair where one path is a portal handle and the other isn't is one file listed against itself under any definition of duplicate you could ever adopt. You don't need to resolve anything to know that. You need a prefix check.

// Inside each candidate cluster:
// if any row has a durable (non-portal) path,
// drop the portal-handle rows — they are aliases,
// not copies.
if group.iter().any(|f| !is_portal_document_path(&f.path)) {
    group.retain(|f| !is_portal_document_path(&f.path));
}

Three properties, each pinned by a unit test: an alias pair vanishes from the results; a genuine duplicate (two real copies) still reports even when a portal alias tags along; and files seen only through the portal keep their rows, because nothing else stands for them. That last clause preserves one honest residual: two portal handles with no durable sibling might still be one file granted twice. Which is why there were experiments.

4Experiment one: the trash button that lied

A sacrificial file, a Delete key, and a toast with no basis in fact.

The open question: if the noisy list ever tricked someone into clicking "trash the other copy" on a portal-handle row, what would actually happen? The pessimistic guess was data loss — trash reaches through the FUSE mount and deletes the only real file, the exact disaster the guard exists to prevent. Five minutes with a sacrificial file settled it:

  1. Make a folder outside every static grant, put two throwaway images in it, open it through the file chooser. It indexes under /run/user/1000/doc/… only.
  2. Select one, press Delete. Then look at the real folder from a host terminal.

Result: the file was untouched. Host trash: empty. The sandbox's private data dir: no trash there either. GIO simply refuses to trash across the document mount. So the catastrophic version is off the table — a portal/portal false pair cannot delete real bytes. Good news, except for what the app was doing while all that nothing happened:

The screenshot from the test shows the image still sitting in the grid, still selected, above a toast reading "Moved 1 image to Trash."

The toast fired when the operation was requested, not when it succeeded — the same shape as an earlier bug where tagging toasted success before the write landed. And the duplicates flow was worse than cosmetic: after requesting the trash, it marked the rows missing in the index regardless of the outcome, then refreshed the list. A failed trash would still make the "duplicate" vanish from the card and stamp the database with a fact that isn't one — missing, on a row whose file is right there.

The fix is the boring, correct plumbing: count the async results, toast what actually moved, toast each failure with its actual error, and mark rows missing only inside the success path. Failure now looks like failure.

The pattern to grep for

Any UI message in the same breath as an async request is a claim about the future. toast(); request(); is a lie waiting for its moment — and the moment will pick the worst possible audience: the user who was already suspicious.

5Experiment two: do handles leak?

If every session mints new paths, the index grows forever.

The second worry was quieter but compounding. Portal handles were assumed to be per-session — the register literally said "opaque per-session document paths". If true, every launch that reopens a folder through the chooser mints fresh handles, which means fresh index rows, which means the portal side of the index grows without bound and every scan re-hashes content the durable rows already cover. That assumption had never actually been tested, and it decides how long the deferred structural fix can stay deferred.

The test is one command from a host terminal, because the document mount is visible outside the sandbox:

$ find /run/user/1000/doc -maxdepth 2 -name 'test-folder'
/run/user/1000/doc/9bf3a2/test-folder

Open the folder through the chooser, note the handle. Quit the app, relaunch, open the same folder through the chooser again, run the command again. Same single handle both times; the indexed row count didn't move. The portal reuses the existing document grant instead of minting a new one — the "per-session" assumption was simply wrong, the index does not leak, and the deferral is safe. One measured fact retired a standing anxiety and corrected the record that had been feeding it.

6Bycatch: the two-file scan that took five minutes

The best bug reports are the ones your own tests file against you.

Experiment two had a confusing middle act. After opening the two-image test folder, the grid showed both thumbnails immediately — and the index showed nothing. Not stale rows: no rows, for minutes, while a query loop watched for them. The indexer thread was pegged the whole time.

The explanation was a known mechanism finally caught in the act. All index work — scans and user writes alike — flows through one worker thread, one request at a time. At launch the app queues a rescan of every library root; on a ~193,000-file index that single request occupies the worker for on the order of ten minutes even when nothing changed. The two-file folder scan sat in the queue behind it for about five. Until then, the folder's images have no content hashes stamped, which quietly degrades everything keyed on them — tagging, drag-to-collection — into "still indexing…" territory. A mystery from an earlier debugging session (five failed attempts to act on a freshly opened folder) matches this exactly.

Nothing was fixed that day — the finding was recorded, re-prioritised, and left deliberately undesigned. But it upgraded from "mechanism that exists" to "measured, with a timeline", and that's usually the step that gets a bug fixed.

7Scope: what a duplicate even is

The last fix wasn't code. It was a definition. Then it was one field.

With the aliases collapsed and the trash flow honest, one design question remained, and it belongs to the owner of the library, not the implementation: the scan still ran over everything. Whole library, always. And at whole-library scale, "duplicate" stops being a technical property:

A copy of the same photo inside the folder you're organising is waste. The same photo on a second drive is not waste — it's a backup, and flagging it for deletion is a duplicate finder recommending you undo your backup strategy.

So the definition Vitrine landed on: duplicates are meaningful within the tree you are looking at. Open a folder, and Find Duplicates scans that subtree; open nothing, and it scans the whole library because you asked it to. The page header says which, in plain words — "in ~/Pictures/trip" — because a scoped answer presented as a global one would be its own little lie. Anything cleverer (volume awareness, "compare these two drives", intent) is deliberately left to a future extension layer rather than grown into the core.

And the implementation cost of the whole product decision? The index's query struct already had an under field — a subtree restriction that compiles to an indexed range scan, built long ago for folder browsing. Scoping Find Duplicates was threading one optional string through two functions, plus a regression test proving a cross-tree pair stops reporting when scoped. The scan gets faster as a side effect, since the BK-tree now builds over the subtree instead of the world.

The short version

The feature was "done" for weeks — clustering correct, BK-tree fast, cards pretty. What made it trustworthy was everything else: refusing to let a hash answer a question about files, a guard for the case that's wrong under every definition, two five-minute experiments instead of two standing assumptions, an error path that tells the truth, and a scope that matches what a human means by the word "duplicate". None of that was hard. All of it was the actual feature.
