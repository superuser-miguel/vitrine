Six bugs in an image browser

A day of debugging Vitrine — a GTK4/Rust image browser with a SQLite catalog — and what each bug turned out to be hiding. Including the ones where the diagnosis was confidently wrong.

July 2026 · GNOME · Rust · GTK4 · SQLite · Flatpak

Vitrine indexes images into SQLite and keys everything — tags, ratings, crops — to a BLAKE3 content hash, so annotations survive renames. The library under test held about 193,000 files.

The reported symptoms were vague in the way real symptoms usually are: drag-and-drop is very bad, tagging is slow, and sometimes it stops working even though it says it worked. Every one of those turned out to be a different bug, and the most useful thing we found wasn't a bug at all.

1Instrumentation that couldn't see its own bugs

A 92,000-line debug log in which none of the bugs appear.

Vitrine has a decent debug HUD: a stats line every second, plus per-item probes, all written to a log. The log from the problem session was 92,375 lines. Here is everything in it:

63065  VDBG-FILMBIND     2296  VDBG-FILM
11767  VDBG-FILL         2038  VDBG-VIEWER
 9648  VDBG-GRIDFILL
 3552  VDBG            (the once-a-second stats line)

Every probe is frame-shaped or decode-shaped. Grepping that log for tag, drag or drop returns zero matches. The instrumentation had been built while chasing scroll performance, and it measured scroll performance beautifully.

So when the UI froze for two and a half seconds, the log recorded this:

stall=2514ms  decode[live=0 done=3894 +0/s]  queued=0  rss=768MB
stall=2541ms  decode[live=0 done=3894 +0/s]  queued=0  rss=768MB
stall=2525ms  decode[live=0 done=3894 +0/s]  queued=0  rss=768MB

Nothing decoding, nothing queued, memory flat, and — critically — not one per-item probe firing during the freeze. The main thread was blocked doing something the log had no vocabulary for. The freezes were recorded as silence.

Two smaller traps in the same log. A frame_max of 22,490ms looked alarming until we noticed it came with stall=3ms — it was the idle gap between redraws, not a hang. And cache_hit=0% throughout looked like a broken cache until we checked the run flags: the session used --cold, which skips cache reads deliberately.

Lesson

Instrumentation records the bugs you had when you wrote it. A log that looks healthy during a reproducible failure is itself the finding — and the first fix, because everything downstream depends on it. Adding four probes (VDBG-TAG, VDBG-DRAG, VDBG-DROP, VDBG-WRITE) went on to disprove two of our leading theories in a single run.

2A tag menu that got slower the more you used it

Cost proportional to tags × files, on the UI thread.

Opening the tag menu froze the app for 2.5 seconds, then 3.4 seconds later the same day. The query behind it counts how many images carry each tag:

SELECT t.id, t.name,
       (SELECT count(*) FROM file_tags ft
        JOIN files f ON f.content_hash = ft.content_hash AND f.missing = 0
        WHERE ft.tag_id = t.id)
FROM tags t ORDER BY t.name COLLATE NOCASE

That reads like an indexed lookup. EXPLAIN QUERY PLAN disagreed:

SCAN t USING COVERING INDEX sqlite_autoindex_tags_1
`--CORRELATED SCALAR SUBQUERY 1
   |--SCAN f                     ← full scan, 192,861 rows
   `--SEARCH ft USING ... (content_hash=? AND tag_id=?)

The file_tags table is keyed PRIMARY KEY (content_hash, tag_id). Filtering on tag_id alone can't use that index — it's the second column. With no usable index, SQLite inverted the loop and drove the subquery from a full scan of every file in the library. Once per tag.

Which makes the cost tags × files. Every tag the user created made the menu slower for the entire library — a performance bug that degrades as the feature succeeds. We confirmed the model by predicting the cost at 9 tags from the measurement at 7:

TagsPredictedMeasuredIn-app freeze
7—0.41s~2.5s
90.53s0.54s~3.4s

The fix was one index:

CREATE INDEX idx_file_tags_tag ON file_tags(tag_id);

The plan flips from SCAN files to SEARCH file_tags, and the query drops from 0.54s to under 10ms. Worst main-loop stall across a session went from 3387ms to 125ms.

There was a second, sillier half. The tag filter dropdown called the same function, then discarded every count and used only the names — paying the full cost for data it threw away.

Lesson

A composite primary key is not an index on each of its columns. And when a slowdown scales with how much the user has invested in a feature, that's worth catching early — the people hurt most are the ones using it most. The regression test asserts the query plan, not the timing, so it fails loudly rather than getting slowly slower.

3Portal paths aren't paths

The bug that turned out to be four bugs.

Vitrine ships as a Flatpak with a deliberately narrow sandbox — read access to ~/Pictures and nothing else. Everything beyond that arrives through the XDG file chooser portal, which hands the app a path like:

/run/user/1000/doc/ZmzOt26RunVwVA8lnYBSqg/Ashley Trevort/utHPx4N.jpg

That is not where the file lives. It's an opaque per-document handle on a FUSE mount, valid while the grant lasts. Vitrine stored it in the path column as though it were an address. So the index ended up with two unrelated families:

Path familyRows
/home/…/Pictures/…79,020
/run/user/1000/doc/…114,048

10,534 content hashes appeared under both — the same bytes, indexed twice, under two names. That single modelling error produced four separate symptoms, none of which looked related when reported:

Tag counts were wrong

files.path is UNIQUE, so files is keyed by path. One image reached two ways is two rows, and the tag count counted rows. A tag on 7 images displayed as 15.

Duplicate detection was 90% noise

"Find duplicates" groups present files by content hash. Portal/real pairs match perfectly, so the feature reported each of them as a duplicate of itself: 41,729 clusters, of which 4,022 were real. The other 90% were one file listed against itself.

And that was nearly destructive

The duplicates card offered "Trash the other copy (keep largest)" — one click, no confirmation. On a portal/real pair, "the other copy" is the same bytes the keeper points at.

Images silently failed to load

The last symptom is the one that makes the model concrete. Some images simply failed to render:

thumbnail file:///run/user/1000/doc/ZmzOt26RunVwVA8lnYBSqg/Ashley Trevort/utHPx4N.jpg
  Failed to load: No such file or directory

The first explanation offered here was that the grant had lapsed. It hadn't — and the correction is more interesting than the guess. Asking the portal what that handle actually points to:

$ gdbus call --session --dest org.freedesktop.portal.Documents \
    --object-path /org/freedesktop/portal/documents \
    --method org.freedesktop.portal.Documents.Info "ZmzOt26RunVwVA8lnYBSqg"

(b'/run/media/definitive_group/873d7a68-…/EasyStore/Pictures/Instagram/Ashley Trevort',
 {'io.github.superuser_miguel.Vitrine': ['read', 'write', 'grant-permissions']})

The grant is intact — the permissions are right there. The handle points at an external USB drive that is currently unplugged. The portal directory still exists and is registered; it is simply empty, because the volume behind it is gone. Across this library, 44,340 indexed files sit behind eleven handles pointing at that one drive.

Which makes "missing" the wrong word for what the app was showing. A file on a disconnected drive is offline — a third state, distinct from both present and deleted, and the only one of the three that resolves itself when you plug the drive back in.

The project owner put it better than the analysis had:

Portal paths should not be treated as a path where files live. It's a symlink.

Exactly right, and it points straight at the fix. Where the same content exists under both a portal handle and a real path, prefer the real one — for display, and for duplicate detection:

pub fn prefer_durable_paths(files: Vec<FileRecord>) -> Vec<FileRecord>

Two constraints make it correct rather than merely effective. It groups by content hash — a near-duplicate cluster mixes different hashes, so dropping portal rows across a whole cluster could discard a genuinely distinct image. And content reachable only through a portal keeps its row, because nothing else represents it.

The portal has more to say about all of this than fits here — how handles are minted, why one folder pick produced 114,048 paths, and how to ask it what a handle really points at. That's its own article.

That's a presentation-layer fix, and it's honest about its limits: it rescues files that have a durable path somewhere. Anything indexed solely through the portal is still one lapsed grant from being unreachable. The real fix — resolving handles to host paths via the Documents portal and storing both — is still on the list.

Lesson

One wrong assumption in the data model surfaced as four unrelated-looking bug reports: a wrong number, a noisy feature, a dangerous button, and a rendering failure. When several symptoms resist a common explanation, the shared cause is often a level below all of them. And the sandbox you chose is part of your data model whether you modelled it or not.

A corollary learned the hard way here: an opaque handle can fail for reasons that have nothing to do with the handle. Before theorising about why an indirection broke, ask the thing that issued it — the portal has an Info call that answers in one command, and it contradicted the guess.

4A UI reporting success it couldn't know

"It says it tagged them, but it didn't."

Annotation writes go to a single writer thread over a channel. Every one of the twelve send sites looked like this:

pub fn tag(&self, name: &str, hashes: &[String], add: bool) {
    let _ = self.requests.try_send(Request::Tag { /* … */ });
}

And the calling UI code like this:

if let Some(indexer) = self.imp().indexer.borrow().as_ref() {
    indexer.annotator().tag(name, &hashes, true);
}
self.toast("Tagged {n} images");        // ← outside the guard, always runs

Three layers of silence with a success message on top. The write could fail, the guard could skip, and the toast fired regardless.

The interesting part was in the channel's type. It's unbounded — which means try_send cannot fail from backpressure. The only way it fails is a dropped receiver, i.e. the writer thread is gone. That made the discarded Result an exact, zero-cost liveness detector, thrown away twelve times.

Wiring it up, plus a queue-depth reading, distinguished two failure modes that had been indistinguishable from the UI: a dead writer (send fails) versus a blocked one (queue grows behind a long scan). They need different fixes.

Then the instrumented run said this:

VDBG-WRITE op=tag_add rows=7 queued=0 accepted=true

Every write accepted, queue never above zero — and the tags were confirmed present in the database. Both theories were wrong. The writes had always worked. "Tagging says it worked but didn't" was the 3-second freeze from §2 making a successful action feel like a failed one.

Lesson

The most valuable instrumentation isn't the kind that confirms your theory — it's the kind that can refute it. Here the refutation was already in the type system: an unbounded channel's send failure has exactly one meaning, and the code discarded it. Also worth stating plainly: never report success above the layer that could fail. If the UI can't know, it shouldn't claim.

5Delete meaning two different things

The engine already had the fix.

In a collection view — a curated set spanning folders — pressing Delete moved the underlying files to the trash. The keybinding had no idea what it was looking at:

(_, Key::Delete) => window.trash_selected(),

Meanwhile the collection's context menu offered "Delete", which deleted the entire collection. So there was no gesture that removed one image from a collection, and the obvious one destroyed originals.

The engine had solved this already:

collections.rs:162  pub fn remove_from_catalog(&self, id: i64, hashes: &[String])
collections.rs:281  (unit test — passing)

Implemented, tested, and reachable from nothing. No request variant, no handle method, no UI. The same pattern showed up twice more: set_catalog_order (unwired) and the tag API, whose add: bool parameter meant it had always supported removal — the UI just passed true everywhere.

Lesson

A passing unit test proves the function works, not that anything calls it. In a layered codebase, "implemented" and "reachable" are different states, and the gap between them is invisible to both the test suite and the type checker. Worth a periodic grep for public engine functions with no callers outside their own tests.

6What the debugging got wrong

This session was done with an AI assistant. Here is where it was confidently wrong.

The write-ups above are tidied. The session wasn't. Three failures are worth recording, because they share one shape.

A fix that was a no-op

When file drops intermittently resolved nothing, the theory was SQLite lock contention: the default busy_timeout is zero, so a reader opened at the wrong moment fails outright. A timeout was added, with a confident comment explaining the bug it fixed.

Then a test was written to prove it. The test passed without the fix. A second, sharper test also passed without it. Finally, the obvious check:

rusqlite default busy_timeout = 5000 ms

rusqlite already sets it. The change did nothing, and the comment asserted a cause that had been ruled out. It was reverted. The real cause is still unknown — but the diagnostics added alongside it will name it next time.

A second no-op, in CSS

Multi-selection in the grid rendered as one saturated slab. Diagnosis: "Adwaita fills a selected item with the solid accent colour; soften it to a 25% tint with a 9px radius." Shipped, described as fixed.

It changed nothing, because Adwaita's own stylesheet already says:

gridview > child          { padding: 3px; border-radius: 9px; }
gridview > child:selected { background-color: color-mix(in srgb, var(--accent-bg-color) 25%, transparent); }

25% softened to 25%. Extracting that stylesheet from the runtime took one command, and was done after shipping the fix rather than before writing it.

Tuning the wrong dimension, three times

Even after that, two more attempts adjusted opacity. The actual cause was layout: 7px of padding sat inside the tinted rectangle (4px from the cell, 3px from the theme) and only 2px outside it, so selected tiles grew until they touched. There were no gaps to soften. No opacity value would have produced one — lowering it just made a lighter unbroken field.

The fix was moving spacing from inside the tint to outside it. The evidence had been sitting in the repository the whole time: a screenshot showing, unmistakably, zero gap between tiles.

The shape

All three failures verified the hypotheses that felt uncertain and skipped the ones that felt obvious — which is backwards. Every skipped check was one command: read the theme, query the default, look at the screenshot. The expensive mistakes weren't hard problems misjudged; they were easy checks not run, on claims that felt too obvious to test. If a fix depends on what some other system already does — a theme, a library default, a portal — read that system before writing the fix, not after shipping it.
