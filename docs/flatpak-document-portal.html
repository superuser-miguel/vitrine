What the document portal actually does

Flatpak's document portal is usually described in one line: it lets a sandboxed app open a file it has no permission to see. That line is true and it hides almost everything that matters. This is a hands-on tour — with numbers from a real 193,000-file library that the portal quietly broke.

July 2026 · Flatpak · xdg-desktop-portal · FUSE · D-Bus

Everything below was measured on one live system, not read from a spec. If you have Flatpak apps installed, every command here will work on yours, and the numbers will be different in interesting ways.

1The problem portals solve

Least privilege collides with "let me open that file".

A well-behaved Flatpak declares the narrowest filesystem access it can. Vitrine declares exactly one static root:

finish-args:
  - --filesystem=xdg-pictures
  # deliberately absent: --filesystem=home / host

Which raises the obvious question: what happens when the user wants to open something that isn't in ~/Pictures? Widening the sandbox defeats the point, and matters more than it looks — an app that later ships helper binaries (ImageMagick, ffmpeg) would extend that grant to them too.

The portal's answer is a nice inversion. The app doesn't get permission to a location; the user's choice creates a permission. You pick a folder in the file chooser — which runs outside the sandbox, in a separate process the app cannot influence — and the portal hands the app a path it can open. The app never gains access to anything the user didn't point at.

That indirection is the whole design, and it's the source of every consequence below.

2Anatomy of a document handle

Not a path. A handle that is shaped like a path.

What the app receives looks like this:

/run/user/1000/doc/ZmzOt26RunVwVA8lnYBSqg/Ashley Trevort/utHPx4N.jpg
└─────┬──────┘ └┬┘ └──────────┬─────────┘ └──────┬─────┘ └────┬────┘
  per-user      │        document id        granted dir     the file
  runtime dir   │        (opaque)           basename
                └── FUSE mount

The mount is real and inspectable:

$ mount | grep /run/user/1000/doc
portal on /run/user/1000/doc type fuse.portal (rw,nosuid,nodev,relatime,user_id=1000,…)

$ ls /run/user/1000/doc | wc -l
759

759 documents on this system. Two things about that directory are worth noticing.

First, the document ID is opaque and stable. It is not derived from the path, and it does not change between sessions — the grants live in a database that survives reboots (~/.local/share/flatpak/db/documents, 176 KB here). A common assumption is that these are ephemeral per-session handles. They aren't, and an app that treats them as throwaway will churn its own state for no reason.

Second, there is a by-app subdirectory:

$ ls /run/user/1000/doc/by-app/
app.devsuite.Ptyxis  ca.desrt.dconf-editor  com.github.flxzt.rnote  com.spotify.Client  …

$ ls /run/user/1000/doc/by-app/io.github.superuser_miguel.Vitrine/ | wc -l
19

The FUSE filesystem is permission-aware per process. An app sees the documents it has been granted and not the rest — which is why the same path can be valid inside one sandbox and meaningless in another, and why debugging this from a host shell can mislead you.

3Directory grants: 11 picks, 114,048 paths

One user gesture, an entire subtree.

The portal grants whole directories, not just single files. When it does, the handle contains the granted directory's basename, and the real tree hangs below it:

host   /home/definitive_group/my-progs/Image_Viewer/docs
handle /run/user/1000/doc/niZCXQRp70wAVxqNSSlm9A/docs/index.html
                                                 └┬─┘
                                        basename of the granted dir

Which means a single "open folder" can mint an unbounded number of valid paths. In the library that prompted all this, the entire portal-side index — 114,048 rows — came from just eleven directory picks:

DocumentFiles indexedPoints at
wz5DlG7x8Fj…42,589~/Inkscape_Projects
my1ebCCAFWt…27,090~/Inkscape_Projects/Jav_avif
ZmzOt26RunV…8,743EasyStore (USB)
hB0wsj5swla…7,940EasyStore (USB)
… 7 more …

Note the first two rows: one grant nested inside another. The portal is perfectly happy to issue overlapping handles, so the same file can be reachable through more than one document ID as well as through a directly-granted root. Nothing de-duplicates that for you.

4The registry, and how to ask it things

The portal will tell you what a handle means. Ask it.

The FUSE mount is only half of it; the other half is a D-Bus service that knows what every handle points to. Introspecting it is the single most useful thing you can do when portal paths are confusing you:

$ gdbus introspect --session \
    --dest org.freedesktop.portal.Documents \
    --object-path /org/freedesktop/portal/documents

Interface version 5 here, with more than the docs led me to expect:

MethodAnswers
Info(doc_id)host path + which apps hold what permissions
Lookup(filename)host path → doc id, the reverse direction
GetHostPaths(ids)batch resolution of many handles at once
List(app_id)every document and its real path
Add / AddNamed / AddFullmint a handle from an fd
GrantPermissions / Revoke / Deletemanage grants

A single call answers "what is this thing really?":

$ gdbus call --session --dest org.freedesktop.portal.Documents \
    --object-path /org/freedesktop/portal/documents \
    --method org.freedesktop.portal.Documents.Info "ZmzOt26RunVwVA8lnYBSqg"

(b'/run/media/…/EasyStore/Pictures/Instagram/Ashley Trevort',
 {'io.github.superuser_miguel.Vitrine': ['read', 'write', 'grant-permissions']})

Two things fall out. Handles are bidirectionally resolvable, so an app that wants a stable identity for a file is not stuck with the opaque form — it can store the host path alongside it. And GetHostPaths makes that cheap in bulk, which matters if you're resolving at index time rather than per file.

The one command worth remembering

Documents.Info(doc_id). When an opaque handle misbehaves, the service that issued it will tell you what it points at and who may touch it. In this investigation that call overturned a diagnosis that had already been written up and published — the guess was "the grant lapsed", the answer was "the grant is fine, the drive is unplugged".

5Three states, not two

Present, deleted — and offline.

The bug that started this: images in the library silently failed to render.

thumbnail file:///run/user/1000/doc/ZmzOt26RunVwVA8lnYBSqg/Ashley Trevort/utHPx4N.jpg
  Failed to load: No such file or directory

"No such file or directory" reads like a deleted file. The portal directory exists, is registered, and is empty:

$ ls -la /run/user/1000/doc/ZmzOt26RunVwVA8lnYBSqg
total 0
dr-x------ 2 …  .
dr-x------ 2 …  ..

And Info explains why: the handle points into /run/media/…/EasyStore/, an external USB drive, and /run/media/definitive_group/ is currently empty. The drive is unplugged. The grant never lapsed — the permissions are still listed, and they will work again the moment the drive is reconnected.

Across this library, 44,340 of 114,048 portal-indexed files — 39% — were sitting behind that one disconnected drive.

So a sandboxed app needs three states where most codebases have two:

StateCauseRight response
Presentresolvesuse it
Offlinevolume unmounted, grant intactkeep the record, show it as unavailable, retry on mount
Gonefile deleted or grant revokedmark missing

The distinction is not pedantry. An indexer that treats offline as gone will happily mark 44,340 files missing because a USB drive was unplugged, and — depending on how aggressive its reconciliation is — discard the tags and ratings attached to them. The kernel gives you the same ENOENT for all three; only the portal and the mount table can tell them apart.

Watch for

Any "reconcile deleted files" pass that runs over portal paths. If it can't distinguish an unmounted volume from a deleted file, an unplugged drive looks exactly like a mass deletion — and reconciliation is usually written on the assumption that the filesystem is telling the truth.

6Why this hurts catalog apps specifically

A viewer opens a file. A catalog remembers it.

If your app opens a file, does something, and forgets it, none of this matters. The portal hands you a path, the path works, you're done.

The trouble starts when you persist the path — and a catalog app persists everything. Vitrine stores one row per file, keyed UNIQUE on path, with tags and ratings hanging off a content hash. Storing handles as though they were locations produced four unrelated-looking bugs:

  • Double-counting. The same image reached through both a granted root and a portal handle is two rows. A tag on 7 images displayed as 15.
  • False duplicates. "Find duplicates" groups by content hash; portal/real pairs match perfectly. 41,729 clusters reported, 4,022 real — 90% noise.
  • A dangerous button. That feature offered "trash the other copy". On a portal/real pair, the other copy is the same bytes.
  • Invisible files. The unplugged-drive case above.

Each was reported separately, none looked related, and all four were the same modelling error: a handle is not an address.

The eventual shape of the fix separates the two questions the path column was trying to answer at once:

  • How do I open this? → the portal handle. It is the only thing that works, and for content outside the granted roots there is no alternative.
  • Is this the same file as that? → the resolved host path, from Info / GetHostPaths, stored alongside.

Content hashing gets you part way — it is why tags survived all of this unscathed — but it answers a third question, "is this the same content", which is not the same as "is this the same file". Two genuine copies of one image share a hash and should stay two rows; one file seen through two routes shares a hash and should be one. Only a resolved host path separates those cases.

7Rules of thumb

  1. Treat a portal path as a handle, not a location. Fine to open, fine to store as how to open. Not an identity, not something to show a user, not something to compare.
  2. Resolve to the host path when you persist. Documents.Info for one, GetHostPaths for many. Store both: the handle to open with, the real path to reason with.
  3. Model "offline" explicitly. Unmounted volume ≠ deleted file, and the syscall won't tell you which. Check the mount table before concluding anything.
  4. Expect overlapping grants. Nested directory grants are normal; one file can be reachable several ways. De-duplicate on something durable.
  5. Don't assume handles are ephemeral. They persist across reboots in a real database. Churning them on every launch is wasted work.
  6. Debug from inside the sandbox. The FUSE view is permission-aware per process; a host shell sees a different world than your app does.
  7. Ask the portal before theorising. One Info call beats an afternoon of inference, and it will occasionally tell you your whole diagnosis was wrong.
The short version

The document portal is an excellent piece of design that trades one guarantee for another: you give up "paths mean places" and you get "the user's choice is the only permission". Apps that hold files briefly never notice the trade. Apps that remember files pay for it in every column where they wrote a handle down and called it an address.
